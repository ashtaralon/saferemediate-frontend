Topology v0.2 · Estate mode (iteration 5) — alon-prod, live data. The counterpart to Triage mode (topology-v0.2.html). Same AWS reference-architecture template, same Neo4j data, same detail panel — but a different operator job. Triage answers "what's the single worst thing in 5 seconds?". Estate answers "scan the whole estate" or "find a specific thing". The visual changes follow that job split:

What changes vs Triage. Headline swaps the prose "worst right now" sentence for a five-tile KPI strip (53 workloads · 7 flagged · 3 stale · 41 unscored · 0% posture fresh). The "0% confidence fresh" tile is the dramatic finding Estate uniquely surfaces — every single posture measurement in this estate is >7d stale, a systemic data-coverage issue Triage can't show because it's not about urgency. Halos drop the pulse and shrink to thin rings — visual presence, not visual emergency, because no node is being spotlighted. Right rail swaps "Next worst" for filters by workload type, severity, VPC, and confidence state (with counts per filter, so operators see the lay-of-the-land at a glance). Detail panel hidden by default; same panel as Triage, opens on node click. Mode toggle top-left flips between the two surfaces — stable position so muscle memory carries.

Why keep the AWS template across both modes. Tempting to flip Estate to a flat taxonomy view (group all EC2s together, all Lambdas together, etc.). Rejected: operators have learned to read the AWS reference architecture, and the value of the template is precisely that it stays the same canvas across views. Different layouts per mode would double the cognitive cost. The mode toggle should change weighting and emphasis, not where things physically sit.

The "Confidence 0% fresh" KPI is the headline story for this estate. Surface 53 workloads, but every single posture verdict is more than 7 days old. That's a system-health finding the Triage view literally cannot express because it's scoped to one spotlighted node. Per the risk-contract's MIN-of-contributors confidence model, this would drive every node's composite confidence down to DEGRADED at best — and Estate mode renders that aggregate fact where it lives, not behind a tooltip.

Multi-VPC handling (Phase 1 of Estate). alon-prod has 2 active VPCs (vpc-0329e985… Payment-Production with 8 workloads, vpc-086bcc21… with 6). v0.2 Estate renders the primary VPC by default with a radio switcher in the rail. Simultaneous multi-VPC layout (stacked frames, side-by-side, or carousel) is a v0.3 / v1 problem — Phase 1 ships with the switcher so the affordance exists even if multi-render doesn't.

Data-trust amber is an audit-trail artifact, not just a UX hygiene signal (iteration 3). The IAM section's stale-count callout is framed loosely as "trust" / "epistemic honesty" — true but soft. For Cyntro's regulated ICP (fintech, healthcare, federal), the stronger argument is regulatory defense. A CISO before an FFIEC examiner or OCC auditor must demonstrate that the inputs to a security decision were verified. "Cyntro flagged stale IAM counts for this role and we resolved the staleness before reducing policy" is a defensible audit trail. "Cyntro silently used stale data and we made the decision off it" is a finding. Amber callouts aren't just informing the operator — they're producing the artifact the operator's compliance team needs. Strategic decks and sales briefs should lead with this framing, not the trust framing.

Amber is a transient self-healing state, never a permanent label (iteration 3). The earliest version of the elaboration warned about decorative ⚠ icons becoming wallpaper, but missed the same failure mode for legitimate amber that never auto-resolves. If "Re-sync IAM counts" sits unclicked for three weeks because the operator forgets and nothing else acts, it becomes background noise — the discipline erodes via its own success. The contract for every amber surface: (a) auto-stale threshold = N min since last successful collector run, (b) auto-resolve when fresh data lands — no operator action on the happy path, (c) operator action surfaces only when collector durably broken (> M min), at which point visual escalates from stale to broken. Spec'd into the IAM sync-warn box above; applies wherever amber is used across the product.

Composite-score data-confidence is the priority propagation point (iteration 3). Five surfaces need data-trust treatment (IAM-strip role chips, headline-strip fallback, auto-remediation gating, view-parity gating, composite-score rollup). They're not equal-priority — the composite score is the single silent-failure point. Every other surface is *visible*; operators see amber and self-correct. The score is a number. If it doesn't degrade when its inputs are stale, four honest panels can aggregate into a dishonest rollup, and the operator catches the contradiction independently. That kills global trust. The risk chip on every node, the headline-strip risk integer, and the right-rail rankings must all derive from a score function that exposes "low confidence — 4 collectors stale" inline, not behind a tooltip. Same architectural pattern as view_parity gating — silent upstream failures undermine honest downstream surfaces.

Risk-score contract — operationalizes the v0.3 principles (iteration 4). docs/topology-v0.2-risk-contract.md defines the JSON contract for a new bulk endpoint /api/topology-risk/{systemName}. Every risk integer on this mockup — headline, node chips, rail entries, detail-panel banner — derives from that contract once implemented. Confidence is a first-class field, computed as MIN(contributors.freshness) rather than averaged — a rollup is no more trustworthy than its weakest input, and that mirrors how a compliance auditor reasons ("did you verify each input?", not "did you verify most inputs?"). Each contributor carries weight + value + freshness metadata + an auto_resolves_when clause that encodes the transient-state contract from iteration 3. Stale workloads exit the score and the rank entirely — they get a separate halo state, not a zero-baseline that would muddy averages. The Score-breakdown section in the detail panel is the inline UI surface that consumes this — operators see contributor bars + freshness chips + auto-resolve clauses without leaving the panel.

What the contract deliberately doesn't bundle. Remediation plans, per-action audit, trend/delta, customer-tunable weights, and cross-system rollup are out of scope for the topology-risk endpoint — each has its own owner (/api/remediation/*, OverrideEvent nodes, a future /trend sibling, PolicyPack, and global-org-score respectively). Keeping the contract narrow is a design decision: every surface that consumes a score wants the same shape, but loading remediation history into the same JSON would force the bulk endpoint into N+1 queries it doesn't need to do.

Honesty caveat. Risk-score integers (87 / 78 / 68 / 65 / 62 / 48) remain illustrative until the canonical node-risk endpoint lands — and per the point above, that endpoint must expose data-confidence as a first-class field, not just an aggregate number. Ranking order is honest, derived from posture_verdict_priority + IAM gap_percentage + jewel flag + stale flag. Auto-Scaling and route-table CIDR rows are AWS-canonical scaffolding shown for layout legibility; Cyntro doesn't score them as nodes today.
