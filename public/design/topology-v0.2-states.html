The states we hadn't designed yet

The Triage and Estate mockups assumed happy-path data. This gallery catalogs the eight non-happy-path states the implementation needs (loading, error, partial, empty, no-selection, stale, race-condition) plus three IAM-Control-Plane-strip variants that handle customer-scale role counts the current 5-role design can't. Each scene has a visual mockup and a design rationale.

Design rationale

Render the topology structure immediately from cached Neo4j metadata (workloads, subnets, SGs, route tables) — that data is already on hand. Don't wait for risk scores to render anything.

Nodes render as skeleton shimmer placeholders, not empty halos. Empty halos read as "all clean" and false-positive the operator into an OK state. Shimmer reads as "computing", correctly.

The toast shows progress through the bulk response (a stream / chunked response could update it incrementally). Estimated time is shown to set expectations — honest about slowness rather than pretending instant.

Anti-pattern: filling the topology with neutral-gray nodes before scores arrive. That looks like the data has answered and the answer is "nothing wrong" — exactly the wrong default.

Design rationale

Never hide the topology — operators need to see what's there even when they can't see what's wrong. The nodes render at 60% opacity to signal "not scored", but they're present and clickable.

The banner gives three options, in priority order:

  • Retry — most common path; backend hiccup, will probably succeed
  • Show last good — pull from use-cached-fetch SWR cache, mark every score with a "from cache · Nm ago" chip per contract §3.5
  • No "Continue without scores" CTA — that would normalize a degraded mode
Carmine banner color is correct here. This is a security tool failing to score security state — that is a security event, even if the failure is transient. Amber would understate it.
State 03

4xx error — system not found

Shows when the operator navigated to a system that no longer exists, was deleted, or was typo'd in the URL.

Design rationale

Route to recovery, not a dead-end error page. Show the system picker pre-populated with fuzzy-matched candidates from the operator's tenant. One click resumes work on the right system.

Each option shows workload count and last-scored time so the operator can verify they're picking a live system, not one that's also broken.

Anti-pattern: "404 — system not found" full-page error. Forces a back-button + nav re-entry. Hostile to muscle memory.

Design rationale

Mixed rendering: scored nodes show halos, unscored nodes show shimmer. The operator immediately understands which findings are real now versus still-coming.

The "Skip slow scorers" CTA lets operators trade completeness for speed when they're triaging under pressure. The skipped contributors get attributed confidence weight = 0 — so the score's confidence field honestly reflects the truncation.

Auto-refresh polls the endpoint every 5s until all nodes return. The implementation uses use-cached-fetch's revalidation, not a fresh fetch — no flash of empty state during refresh.

Critical: the unscored nodes must NOT be assigned a default score (e.g., 0 or 50). That would pollute the rank order in the right rail. They're excluded from rank until they return.
State 05a

Empty system — Cyntro hasn't observed any AWS resources

Shows when a system exists in Cyntro but no collectors have run yet — typical immediately after tenant onboarding.
🌥

No AWS resources observed yet for alon-prod

Cyntro's discovery collectors haven't returned data for this system. Typical first-run takes 6 - 12 minutes from connection.

Last collector activity: none in last 24h.

Design rationale

Empty state is an opportunity moment — operator is new, needs orientation. The card explains what's missing, why, and a single action to resolve.

"Run discovery now" triggers an out-of-band collector dispatch. If automatic discovery is already running on cadence, the CTA reads "Speed up next scan" instead.

The "last collector activity" line is load-bearing. Empty state could mean: (a) tenant is brand new — wait, (b) collectors broken — escalate, (c) collectors running but slow — wait. Surfacing the activity timestamp disambiguates without making operator do detective work.
State 05b

Empty VPC — system has VPCs but none with workloads

Shows when the operator selected a VPC in Estate mode (or one VPC system in Triage mode) that has subnets but no observed compute / database workloads.
VPC · 172.31.0.0/16
⌖

No workloads observed in this VPC

VPC has 4 subnets and 3 route tables, but no EC2, Lambda, RDS, ECS, or other compute observed in the last 7 days. May be a transit-only VPC.

View 4 subnets →

Design rationale

Different from empty-system. The infrastructure is observed (subnets, route tables); just no workloads. This is a legitimate architectural pattern (transit VPCs, default-VPC-leftover, isolated test VPCs).

The "View 4 subnets" link lets the operator inspect what does exist without forcing them to switch views. The topology canvas isn't blank — it's framed by the VPC border, and the empty-state card sits inside it.

If the user's filter rail (Estate mode) is excluding workload types that do exist in this VPC, the card adjusts: "Filter excludes 12 workloads. Adjust filters to see them."

Race condition — clicked node missing from bulk response

Shows when the operator clicks a node before the bulk endpoint has returned its score, or when the clicked node was just removed from AWS between topology render and click.

Computing score…

Workload identified from cached Neo4j metadata. Risk score and contributors requested from /api/topology-risk/{system}/node/{id}.

Design rationale

Optimistic UI: open the panel immediately with the node's identity (name, subnet, IDs) since that comes from cached topology data. Show "Computing score" for the parts that depend on the bulk response.

If the per-node fetch returns a score, the panel fills in. If it returns 404 (node was deleted between topology render and click), the panel switches to a dedicated "This workload was just removed" view with timestamps from CloudTrail.

Anti-pattern: blocking the panel open until the score is ready. Operators experience that as a click that did nothing — they click again and get the inverse race.

Design rationale

Affordance, not invitation. The hint shows the operator what's coming (a list of contents — score, IAM, traffic, remediation, related) rather than decoratively asking them to do something.

The "Top of estate tail" mini-list bridges the gap between "look at the whole map" (Estate mode's job) and "go investigate something specific" (Triage mode's job). Operators who scanned the estate and want to drill in get a starting point without flipping modes.

Why not collapse the panel entirely. Reserving the panel space keeps the layout stable — when the operator does click a node, the panel doesn't shove the topology around to make room.

Shows when the contract response has scored_at older than the operator's freshness threshold (default 1 day). The data is technically valid; it's just confidently old.

Every node halo demoted to amber regardless of underlying severity — global confidence has dropped below the carmine threshold.

Design rationale

This is the amber-as-transient principle (feedback_amber_must_self_heal) at the global scope. The banner declares a self-healing state and the system is already acting on it ("Re-running collectors" — not a button for operator).

Per the contract's MIN-of-contributors confidence model, system-wide stale posture forces every node's confidence to DEGRADED at best. Visually: all halos shift to amber, the underlying severity tier is shown numerically next to the node but de-emphasized.

The hourglass glyph (⌛) next to each score is the inline cue that this number is the stale one. Same affordance everywhere — chip, rail, panel.

"Force re-score" is the operator escape hatch — it triggers out-of-band collector dispatches in priority order (posture first, then IAM, then traffic). The button is the only operator action surfaced in this state.

The current v0.2 design renders 5 roles cleanly. Customer-scale VPCs attach 40 - 200+ roles. This section catalogs the three rendering variants the implementation must handle.

Design rationale

Group by service principal first, not by gap tier or by role name. Customers organize their roles by what kind of workload assumes them — that's how they reason about IAM, that's how they name their roles, that's how they delegate ownership. Severity sorting within a group.

Always show critical-gap roles inline. Even at 247 roles, the 5 - 30 critical ones live above the fold of their group, no expand required. Clean and stale roles collapse into a single "+ N more" link per group.

Service-linked roles are explicitly skipped. Cyntro doesn't modify AWS-managed roles; they have their own least-privilege owned by AWS. Surfacing them just adds noise.

Cross-account / federated as a separate category. Audit-only because Cyntro can't remediate roles trusted from outside the account. Marking them prevents accidental remediation attempts.

Same collapse pattern across all categories. Operator learns one interaction model and it scales from 5 roles to 247 with no new vocabulary.

What this gallery doesn't yet cover

Honest about the design gaps. None are blocking the v0.2 Triage / Estate ship, but they're worth knowing before the backend impl PR lands or the design ports into components/attack-paths-v2/topology-view.tsx.
  • Mobile / narrow viewport. KPI strip's 5-tile grid breaks below ~960px. Filter rail probably needs to collapse to a bottom sheet.
  • Accessibility primitives. No ARIA roles specced. Keyboard navigation through topology + panel + filter rail is undefined.
  • Color-blind safety. Severity uses halo + ring + chip — partially redundant but not fully color-blind safe (carmine + amber are close in protanopia simulation).
  • Serverless / event-driven architectures. AWS reference template assumes Web/App/DB tiers. SAM apps, EventBridge-driven systems, k8s topologies need separate templates.
  • Multi-VPC simultaneous render (deferred from Estate v0.2 to v0.3). Stacked / side-by-side / carousel — three options, none designed.
  • Cross-account topologies. Cyntro's tenant boundary is single-account per system today. Multi-account aggregation is a follow-up.
